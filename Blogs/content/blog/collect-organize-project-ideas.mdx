# How to Collect and Organize Project Ideas at Scale

You opened submissions and got 500 projects in the first week. Now what?

Manual organization = chaos. Here's the systematic approach.

## Submission Form Design

**What to ask:**
```
Required:
- Project title
- Project description (500 words)
- Problem statement
- Proposed solution
- Expected impact
- Team member names

Optional:
- GitHub link / demo
- Video pitch
- Files/documents
- Additional notes
```

**Structure matters:**
- Order questions logically (what → why → how → who)
- Mark required vs. optional clearly
- Provide examples ("Problem: Patients wait 3+ hours...")
- Set field limits (description max 500 words; encourages clarity)

## Organization System

**As submissions arrive:**

### **Step 1: Auto-Categorization**
- Submissions assigned project ID (PROJ-001, PROJ-002)
- Auto-categorized by keywords (AI, Web, Mobile, Hardware, etc.)
- Timestamp recorded
- Submission status tracked (complete, incomplete, flagged)

### **Step 2: Initial Screening**
- Check: Is this complete? (All required fields present)
- If incomplete: Send email "Your submission is missing [field]. Please provide by [DATE] to be considered."
- If complete: Move to review queue

### **Step 3: Categorize by Track** (if applicable)
- Manual or automated (based on project description)
- Tag: "AI/ML", "Healthcare", "Climate", etc.
- Use tags so a project can be in multiple categories

### **Step 4: Organize for Judges**
- Create judge-facing dashboard
- Each judge sees only their assigned projects
- Show: Title, description, team, GitHub, video
- Rubric embedded for scoring

## Tools for Organization

**Spreadsheet (Google Sheets)**
- Pro: Simple, free, familiar
- Con: Doesn't scale past 200 projects; manual; error-prone
- Best for: &lt;200 projects, informal

**Airtable**
- Pro: Database + views, filterable, collaborative
- Con: Still manual entry, clunky at 500+ projects
- Best for: 200-500 projects, semi-professional

**Specialized platform (FairArena, Submittable)**
- Pro: Auto-workflows, submission tracking, judge integration
- Con: Cost ($200-$5,000)
- Best for: 500+ projects, professional event

## Submission Tracking Dashboard

**What organizers need to see:**
- Total received (should match deadline strategy)
- Breakdown by category (is distribution balanced?)
- Completion rate (% with all required fields)
- Submission rate over time (early surge vs. last-minute rush?)
- Flagged submissions (incomplete, disqualified, duplicates)

**Example:**
```
Total: 487 projects
- Complete: 423 (87%)
- Incomplete: 64 (13%) - follow up due by Friday
- By category:
  - AI/ML: 142
  - Healthcare: 89
  - Climate: 156
  - Web: 78
  - Other: 22
```

This data helps you:
- Identify gaps (too few climate projects? More marketing needed)
- Plan judging (allocate judges proportional to project volume)
- Spot issues (incomplete submissions = unclear form?)

## Handling Duplicates & Invalid Submissions

**Duplicate detection:**
- Similar titles + team members = likely duplicate
- Email team: "We received similar submissions. Which one is official?"
- Keep only one; disqualify other

**Invalid submissions:**
- Off-topic (not related to hackathon theme)
- Inappropriate content
- Spam

**Disqualification email:**
- Polite, clear reason why
- Example: "Your project X is outside our competition scope (environmental focused). We appreciate your interest; apply next year!"

## Organizing for Blind Review

**If doing blind review:**
1. Strip identifying information (names, company, etc.)
2. Assign anonymized ID (PROJ-001, etc.)
3. Store mapping separately (PROJ-001 → Team Name) encrypted
4. Judges see only anonymized projects
5. Reveal identities post-judging

**Tools that handle this:**
- Most modern judging platforms auto-anonymize
- Manual: Remove names; find-replace with IDs; maintain encrypted mapping

## Quality Control

**Before sending to judges:**
- [ ] Sample 10 random submissions; verify completeness
- [ ] Check for duplicates
- [ ] Verify categorization (does project actually match its category?)
- [ ] Confirm judge assignments (right judges getting right projects)

**Spot-check reduces errors.**

## Communication to Submitters

**Upon submission:**
- Auto-email: "Received your project X. Judging begins [DATE]."

**Weekly status email:**
- "X projects received. Judging ends [DATE]."
- "Questions? Check our FAQ."

**Pre-announcement:**
- "Results coming [DATE]. Here's the judging criteria."

**Post-results:**
- "You were [ranked #47]. Here's feedback from judges."

## Common Issues & Solutions

**Issue: Too many incomplete submissions**
- Root cause: Form unclear
- Fix: Clarify form; send reminders with examples

**Issue: Duplicates everywhere**
- Root cause: Easy to submit twice; unclear instructions
- Fix: Add"confirmation page" + email deduplication

**Issue: Off-topic submissions**
- Root cause: Theme unclear
- Fix: Put italicized, emphasized theme at top of form

**Issue: Judges say projects are categorized wrong**
- Root cause: Auto-categorization algorithm poor
- Fix: Do manual categorization or train algorithm better

## Organizing for Success

**Timeline:**
- 6 weeks before event: Form ready, tested
- 4 weeks before: Open submissions
- 2 weeks before: Deadline
- 1 week after: Judging begins

**At each stage:**
- Track volume, completeness, categories
- Spot issues early
- Fix form/process if needed

## Conclusion

Collecting and organizing 500+ projects requires system thinking:
1. **Well-designed form** (clear, organized, required/optional clear)
2. **Tracking dashboard** (see volume, completeness, distribution)
3. **Organization workflows** (categorization, deduplication, flagging)
4. **Judge-ready structure** (judges see only what they need)
5. **Communication** (automate updates)

Do this well and judging becomes easy. Miss it and chaos reigns.

**Before your next submission window**: Template your form. Test it yourself. Run through entire workflow once solo. Then open to public.
