# DevSecOps Tutorial: Securing the Software Supply Chain in 2026

DevSecOps is no longer optional in 2026. Software supply chain attacks have exploded: SolarWinds, Log4Shell, left-pad.

DevSecOps = "security shift left"—catch vulnerabilities early, before they reach production.

## What is DevSecOps?

Traditional flow:
```
Code → Build → Test → Deploy → Discover bugs in production
```

DevSecOps:
```
Code → [Scan for secrets] → Build → [Scan dependencies] → 
Test → [SAST/DAST] → Deploy → [Monitor for intrusions]
       ↓ Find 90% of issues before production
```

## The Supply Chain Problem

Your app depends on hundreds of packages:

```
npm install next
  ↓
Installs 100+ transitive dependencies
  ↓
One of them has a known CVE
  ↓
Attacker exploits your app
```

**Solution**: Automate scanning at every step.

## DevSecOps Tools in 2026

### 1. Secrets Scanning

Find hardcoded passwords before they're leaked.

**Tool: Snyk Secrets**

```bash
# Install
npm install -D snyk

# Scan for secrets
snyk secret scan
# Found 2 secrets:
# - AWS_SECRET_ACCESS_KEY in .env
# - STRIPE_SECRET_KEY in config.js
```

**GitHub Native**:
```yaml
# .github/workflows/secrets.yml
name: Secrets Scanner

on: [push, pull_request]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: TruffleHog
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: ${{ github.event.repository.default_branch }}
```

### 2. Dependency Vulnerabilities

Scan your packages for known CVEs.

**Tool: Snyk SCA**

```bash
# Scan dependencies
snyk test

# Output:
# Found 5 vulnerabilities:
# - log4j 2.13.0 (CRITICAL - CVE-2021-44228)
# - moment 2.29.0 (HIGH - ReDoS)
# - lodash-es 4.17.20 (MEDIUM - Prototype pollution)
```

**GitHub Native**:
```yaml
- name: Dependency check
  run: npm audit
  # Fails if vulnerabilities found
```

**Automated Fixing**:
```bash
# Snyk auto-creates PRs with fixes
snyk fix
# Creates PR: "fix: upgrade log4j to 2.18.0"
```

### 3. SAST (Static Application Security Testing)

Find security bugs in your code.

**Tool: SonarQube**

```yaml
# .github/workflows/sonarqube.yml
on: [push, pull_request]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: SonarScanner
        uses: SonarSource/sonarcloud-github-action@master
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
        with:
          args: >
            -Dsonar.projectKey=my-project
            -Dsonar.organization=my-org
```

**Detects**:
- SQL injection
- XSS vulnerabilities
- Hardcoded secrets
- Cryptography issues
- Authentication flaws

### 4. Container Scanning

Find vulnerabilities in Docker images.

**Tool: Trivy**

```bash
# Install
brew install aquasecurity/trivy/trivy

# Scan image
trivy image myapp:latest

# Output:
# libc (CRITICAL - CVE-2021-1024)
# openssl (HIGH - CVE-2020-1971)
```

**GitHub Actions**:
```yaml
- name: Container scanning
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    format: 'sarif'
    output: 'trivy-results.sarif'

- name: Upload to GitHub Security
  uses: github/codeql-action/upload-sarif@v2
  with:
    sarif_file: 'trivy-results.sarif'
```

### 5. DAST (Dynamic Application Security Testing)

Test running application for vulnerabilities.

**Tool: OWASP ZAP**

```yaml
- name: DAST scan
  run: |
    docker run -t owasp/zap2docker-stable zap-baseline.py \
      -t http://localhost:3000 \
      -r zap-report.html
```

Floods your app with attack payloads. Detects:
- SQL injection
- XSS
- CSRF
- Broken auth
- Insecure deserialization

## Complete DevSecOps Pipeline

```yaml
# .github/workflows/devsecops.yml
name: DevSecOps Pipeline

on: [push, pull_request]

jobs:
  secrets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Scan for secrets
        uses: trufflesecurity/trufflehog@main
  
  dependencies:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - uses: actions/setup-node@v3
      
      - name: Scan dependencies
        run: npm audit --audit-level=moderate
  
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: SonarScanner
        uses: SonarSource/sonarcloud-github-action@master
  
  build:
    needs: [secrets, dependencies, sast]
    runs-on: ubuntu-latest
    steps:
      - uses: docker/build-push-action@v4
        with:
          push: true
          tags: myapp:${{ github.sha }}
  
  container-scan:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Trivy scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
  
  deploy:
    needs: container-scan
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - run: kubectl set image deployment/app app=myapp:${{ github.sha }}
  
  dast:
    needs: deploy
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: DAST scan
        run: |
          docker run owasp/zap2docker-stable \
            zap-baseline.py -t https://myapp.com
```

## Policy as Code

Enforce security standards:

```yaml
# policies.rego (OPA Rego language)
package kubernetes

deny[msg] {
  input.request.kind.kind == "Pod"
  image := input.request.object.spec.containers[_].image
  not startswith(image, "docker.io/myregistry/")
  msg := sprintf("Image not from approved registry: %v", [image])
}

deny[msg] {
  input.request.kind.kind == "Pod"
  container := input.request.object.spec.containers[_]
  not container.securityContext.runAsNonRoot
  msg := "Container must run as non-root"
}
```

Deployed in CI/CD:
```bash
conftest test pod.yaml -p policies.rego
# FAIL: pod.yaml - Image not from approved registry
```

## SBOM (Software Bill of Materials)

Catalog every dependency. Required by regulators.

```bash
# Generate SBOM
syft myapp:latest --output spdx > sbom.json

# Output:
{
  "SPDXID": "SPDXRef-Package-myapp",
  "name": "myapp",
  "components": [
    {
      "type": "library",
      "name": "next",
      "version": "14.0.0",
      "purl": "pkg:npm/next@14.0.0"
    },
    ...
  ]
}
```

Upload to tracking system:
```bash
curl -X POST https://sbom-tracking.mycompany.com/upload \
  -H "Authorization: Bearer $TOKEN" \
  -d @sbom.json
```

## Hardening OS Base Images

Never use `FROM ubuntu:latest`. Use hardened images:

```dockerfile
# Bad: Bloated, unpatched
FROM ubuntu:latest

# Good: Minimal, hardened
FROM distroless/nodejs20-debian12

# Or: Security-focused
FROM chainguard/node:latest
```

Compare sizes:
```
ubuntu:latest           77 MB  (many potential vulns)
node:20                 1.1 GB
distroless/nodejs:20    200 MB
chainguard/node         100 MB (hardened, minimal)
```

## Runtime Security

Monitor running apps for attacks:

```yaml
# Falco rules (detect suspicious behavior)
- rule: Unauthorized Process
  condition: >
    spawned_process and 
    container and 
    not proc.name in (acceptable_procs)
  output: Unauthorized process spawned
  priority: WARNING

- rule: Write to System Dirs
  condition: >
    open and 
    container and 
    write and 
    fd.name startswith /etc/
  output: Attempted write to /etc/ in container
  priority: ERROR
```

Deploy in Kubernetes:
```bash
helm install falco falcosecurity/falco \
  --values falco-values.yaml
```

## Cost Estimation

**Self-hosted:**
- SAST (SonarQube): $5k–10k setup, $500/month
- DAST (ZAP): Free, ~100 hours setup
- Dependency scanning: $0 (npm audit), or $200/month (Snyk)
- Container scanning:$0 (Trivy), or $500/month (Aqua)

**Managed SaaS:**
- Snyk: $200–2000/month
- Sonarcloud: $50–500/month
- Aqua: $1000+/month

A good start: Free tools (npm audit, Trivy, TruffleHog) + Snyk ($200/mo).

## Security Checklist

- [ ] Scan for secrets in CI/CD
- [ ] Scan dependencies (npm audit, Snyk)
- [ ] SAST scanning (SonarQube)
- [ ] Container image scanning (Trivy)
- [ ] No root containers (runAsNonRoot: true)
- [ ] Network policies (Kubernetes)
- [ ] Least privilege access (RBAC)
- [ ] Secrets in secure store (Vault, AWS Secrets Manager)
- [ ] Regular backups (tested)
- [ ] Incident response plan
