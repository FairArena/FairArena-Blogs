# How to Manage Remote Hackathon Teams: Ultimate Guide

Running a remote hackathon is different from in-person. You can't walk around observing; you need intentional communication. Here's how to make it work.

## Pre-Event Communication

**Send 2 weeks before:**
- Event schedule (timezone-friendly start time)
- Discord/Slack link (where all communication happens)
- GitHub + deployment checklist
- Judge rubric (so teams know what to build toward)
- FAQ document (anticipate common Q's)

**Send 3 days before:**
- "Event starts Friday 6 PM EST. Here's the link. Show up 15 min early."
- Confirm all team members have access to platforms
- Test: "Can you access Discord? GitHub? Deployment environment?"

## Kickoff Strategy

**Online kickoff (live video, 1 hour):**
- Welcome & problem statement (5 min)
- Demo of submission platform (10 min)
- Judge introductions (10 min)
- Networking breakout rooms (team formation) (20 min)
- Q&A (15 min)

Record the kickoff; post recording for timezone-delayed teams.

## During the Hackathon

**Communication channels:**
- **#announcements**: Major updates (deadline extended, help request, etc.)
- **#general**: Casual chat, questions
- **_#help-technical**: Technical troubleshooting
- **#showcase**: Teams sharing progress

**Communication cadence:**
- Daily check-in message: "Day 2 of 3. Keep shipping! Submit by Sunday 6 PM."
- Respond to help requests within 1 hour (during event hours)
- No radio silence. Unclear = participants worry they're forgotten

**Office hours:**
- Set 2-3 hour "office hours" per day where organizers are available
- Post Zoom link in #help-technical
- Teams can drop in with questions
- Even if nobody joins, being available matters

**Progress tracking:**
- Lightweight daily update: "Teams coding or stuck?"
- Check: Any teams radio silent? Reach out individually.
- Use GitHub as signal (commit activity = building; no commits = stuck)

## Avoiding Common Remote Failures

### **Problem 1: Timezone Hell**
- 24-hour hackathons assume everybody's awake together
- **Solution**: Extended window (48-72 hours) so teams in different zones can participate

### **Problem 2: Unclear Deadlines**
- "Sunday evening" means different things in different timezones
- **Solution**: "Sunday 6 PM EST (all timezones converted on website)"

### **Problem 3: No Sense of Community**
- In-person, energy is contagious. Virtual, it's quiet.
- **Solution**: Forced networking (breakout rooms), team showcases, judge Q&A

### **Problem 4: Submission chaos**
- Teams unsure if they submitted correctly
- **Solution**: Auto-confirmation email immediately + dashboard showing submission status

### **Problem 5: Judge availability**
- Judges time their review for UTC times; some miss window
- **Solution**: Async judging (judges review on their schedule over 1 week, not live)

## Best Practices

**Before Hacking Starts**
- [ ] Test platform (you submit a test project)
- [ ] Test Discord/Slack (channels created, permissions set)
- [ ] Confirm judges have platform access
- [ ] Confirm all team members can access tools

**During Hacking**
- [ ] Check Discord every 2-3 hours for questions
- [ ] Post daily encouragement messages
- [ ] Celebrate early submissions (even if incomplete)
- [ ] Watch for silent teams; reach out

**Submission Day**
- [ ] Reminder email 24 hours before deadline
- [ ] Reminder post in Discord 6 hours before
- [ ] Reminder 1 hour before deadline
- [ ] Monitor submissions in real-time (tracker dashboard)

**Post-Submission**
- [ ] Auto-send confirmation email
- [ ] Thank participants; set judging timeline
- [ ] Share results within 1 week

## Virtual Judging (Async Approach)

**Why async works for remote:**
- Judges in different timezones
- No need to coordinate live panel
- Judges have time to really think

**Process:**
1. All projects submitted by Sunday 6 PM
2. Email goes to judges Monday: "Review these X projects by Friday"
3. Judges evaluate async (their own schedule)
4. Platform auto-calculates by Friday
5. Winners announced Saturday

**Judges see:**
- Project description + GitHub link
- Demo video (pre-recorded)
- Rubric to fill out online
- Progress bar (X of Y projects reviewed)

This works great for remote.

## Common Questions

**Q: How do we prevent cheating?**
A: GitHub commit history shows who actually coded. Ask teams how it works in Q&A.

**Q: Can we do live judging / demo?**
A: Yes, but hard across timezones. Pre-recorded demos + async judging is better UX.

**Q: What if a team doesn't submit?**
A: Email reminder 30 min before deadline. If still no submission, they're out. No exceptions.

**Q: How do we build community if it's remote?**
A: Breakout networking rooms at kickoff, team showcases mid-event, judge Q&A post-event.

## Conclusion

Remote hackathons succeed with:
1. **Clear communication**: Repeated, across channels, timezone-aware
2. **Accessible platforms**: Tech works; documentation is clear
3. **Intentional community**: Breakout rooms, showcases, recognition
4. **Async-first judging**: Single judging window creates timezone nightmares
5. **Responsive organizers**: Available during event; respond quickly to help

Don't just take in-person hackathon and move it online. Redesign for async/remote first. You'll get better participation and better outcomes.
