# Docker Tutorial for Beginners: From Zero to Containers in 2026

Docker is the standard for containerizing applications in 2026. Every job posting for DevOps, backend, or full-stack roles mentions Docker. If you haven't learned it yet, now is the time.

This guide assumes zero Docker knowledge. By the end, you'll understand how containers work, build your first Docker image, and deploy a containerized app.

## What is Docker?

**Docker** is a containerization platform that packages your entire application—code, dependencies, runtime, config—into a single unit called a **container**.

Think of it like shipping containers for apps. Instead of shipping fragile goods loose (where they might break), you pack everything into a standardized container. The container works the same whether it's shipped on a boat, truck, or train.

Same with Docker: your app works the same on your laptop, your teammate's laptop, staging, and production.

## Virtual Machines vs. Containers

**Virtual Machines (VMs):**
- Install a full OS (Windows, Linux) for each app
- Heavy: 500MB–2GB per instance
- Slow to start: minutes
- Good isolation, but resource-intensive

**Containers:**
- Share the host OS kernel
- Lightweight: 10–100MB per instance
- Fast to start: milliseconds
- Good isolation, much more efficient

Example: A VM like VirtualBox or VMware simulates an entire computer. Docker containers share the OS but isolate packages and runtime.

**Result**: You can run 10 containers on the same hardware that might slow down with 2 VMs.

## Key Docker Concepts

### 1. Image

A **Docker image** is a blueprint. Think of it as a snapshot of your entire application: code, dependencies, OS, configuration.

You don't run images directly. You use them to create containers.

### 2. Container

A **container** is a running instance of an image. If an image is a class, a container is an object.

One image can create multiple containers. Each runs independently.

### 3. Dockerfile

A **Dockerfile** is a recipe for building an image. It describes:
- Start from a base image (e.g., `node:20`, `python:3.11`)
- Copy your code
- Install dependencies
- Expose ports
- Define startup command

### 4. Registry

A **registry** is where images are stored (like GitHub for Docker).

**Docker Hub** is the default public registry. You push images there; others pull them.

## Your First Dockerfile

Let's build a simple Node.js app container.

### Step 1: Create a Project

```bash
mkdir docker-app
cd docker-app
npm init -y
npm install express
```

### Step 2: Create an App

**app.js**:
```javascript
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('Hello from Docker!');
});

app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

### Step 3: Create a Dockerfile

**Dockerfile**:
```dockerfile
# Start from official Node image
FROM node:20-alpine

# Set working directory inside container
WORKDIR /app

# Copy package files
COPY package*.json ./

# Install dependencies
RUN npm install

# Copy app code
COPY . .

# Expose port
EXPOSE 3000

# Run the app
CMD ["node", "app.js"]
```

Breaking it down:
- `FROM node:20-alpine`: Use the lightweight Alpine Linux image with Node 20 pre-installed
- `WORKDIR /app`: All commands run inside `/app` in the container
- `COPY package*.json ./`: Copy package.json and package-lock.json (if exists)
- `RUN npm install`: Install dependencies inside the container
- `COPY . .`: Copy your source code
- `EXPOSE 3000`: Your app listens on port 3000
- `CMD ["node", "app.js"]`: Command to run when container starts

### Step 4: Build an Image

```bash
docker build -t my-app:1.0 .
```

Breakdown:
- `docker build`: Build an image from a Dockerfile
- `-t my-app:1.0`: Tag the image as "my-app" with version "1.0"
- `.`: The Dockerfile is in the current directory

Output:
```
[1/7] FROM node:20-alpine
[2/7] WORKDIR /app
[3/7] COPY package*.json ./
[4/7] RUN npm install
[5/7] COPY . .
[6/7] EXPOSE 3000
[7/7] CMD ["node", "app.js"]
=> => naming to docker.io/library/my-app:1.0
```

### Step 5: Run Your Container

```bash
docker run -p 3000:3000 my-app:1.0
```

Breakdown:
- `docker run`: Create and start a container from an image
- `-p 3000:3000`: Map port 3000 from container to your machine
- `my-app:1.0`: The image to run

Output:
```
Server running on port 3000
```

Open `http://localhost:3000` in your browser. You see "Hello from Docker!"

## Essential Docker Commands

```bash
# List all images
docker images

# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# Stop a container
docker stop <container-id>

# Remove a container
docker rm <container-id>

# Remove an image
docker rmi <image-id>

# View logs
docker logs <container-id>

# Execute a command in a running container
docker exec -it <container-id> bash

# Push image to Docker Hub
docker push username/my-app:1.0

# Pull image from Docker Hub
docker pull username/my-app:1.0
```

## Docker Compose (Multiple Containers)

Real apps need a database, cache, queue. Managing multiple containers manually is painful.

**Docker Compose** orchestrates multiple containers with a single YAML file.

### Example: Node App + PostgreSQL + Redis

**docker-compose.yaml**:
```yaml
version: '3.8'

services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/myapp
      - REDIS_URL=redis://cache:6379
    depends_on:
      - db
      - cache
    volumes:
      - .:/app

  db:
    image: postgres:15-alpine
    environment:
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
      - POSTGRES_DB=myapp
    volumes:
      - db_data:/var/lib/postgresql/data

  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  db_data:
```

### Run Everything

```bash
docker-compose up
```

This starts all three services. Your app can connect to `db:5432` and `cache:6379` automatically. Docker's internal networking handles it.

### Stop Everything

```bash
docker-compose down
```

## Best Practices

### 1. Use Lightweight Base Images

```dockerfile
# Good: Alpine is ~50MB
FROM node:20-alpine

# Bad: Full image is 400MB+
FROM node:20
```

### 2. Layer Caching

Order matters. Put frequently-changing code last:

```dockerfile
# Good: dependencies first (rarely change), code last (changes often)
COPY package.json .
RUN npm install
COPY . .

# Bad: code first, dependencies last (rebuilds deps every time)
COPY . .
RUN npm install
```

### 3. Don't Run as Root

```dockerfile
RUN useradd -m appuser
USER appuser
```

### 4. Use .dockerignore

Like `.gitignore`, but for Docker:

```
node_modules
.git
.env
```

### 5. Security: Don't Hardcode Secrets

```dockerfile
# Bad
ENV DATABASE_PASSWORD=secret123

# Good: Pass at runtime
# Then: docker run -e DATABASE_PASSWORD=secret123
```

## Troubleshooting

### Container exits immediately
Check logs:
```bash
docker logs <container-id>
```

Usually the startup command crashed.

### Port already in use
```bash
docker run -p 8000:3000 my-app:1.0
```
Maps port 8000 on your machine to 3000 in container.

### Slow builds
Use a `.dockerignore` file. Exclude `node_modules`, `.git`, etc.
