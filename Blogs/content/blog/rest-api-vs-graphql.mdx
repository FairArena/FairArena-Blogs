# REST API vs GraphQL: Which Should You Choose in 2026?

In 2026, the debate between REST and GraphQL is settled: **both coexist**. REST dominates simple, public APIs. GraphQL powers complex, data-intensive applications.

This guide explains both architectures, their tradeoffs, and when to use each.

## REST API (Representational State Transfer)

### What is REST?

REST is an architectural style using HTTP methods (GET, POST, PUT, DELETE) to perform CRUD operations on resources.

**Example: Blog API**

```
GET    /api/posts          → List all posts
GET    /api/posts/1        → Get post #1
POST   /api/posts          → Create post
PUT    /api/posts/1        → Update post #1
DELETE /api/posts/1        → Delete post #1
GET    /api/posts/1/comments → Get comments for post #1
```

### REST Example Code

**Backend (Node.js/Express)**:
```typescript
app.get('/api/posts', async (req, res) => {
  const posts = await db.post.findMany();
  res.json(posts);
});

app.post('/api/posts', async (req, res) => {
  const post = await db.post.create({ data: req.body });
  res.json(post);
});

app.get('/api/posts/:id', async (req, res) => {
  const post = await db.post.findUnique({
    where: { id: parseInt(req.params.id) },
    include: { author: true, comments: true }
  });
  res.json(post);
});
```

**Frontend (React)**:
```typescript
// Fetch a post with author and comments
const post = await fetch('/api/posts/1').then(r => r.json());
// Returns full post object with author and comments
```

### REST Pros

- **Simple**: Standard HTTP verbs, easy to understand
- **Caching**: HTTP caching works out of the box
- **Tooling**: Postman, curl, browser dev tools
- **Stateless**: No sessions; good for horizontal scaling
- **Standard**: Everyone knows how to build/consume REST APIs

### REST Cons

- **Over-fetching**: API returns extra data you don't need
- **Under-fetching**: Need multiple requests to get related data
- **Versioning**: `/v1/posts` vs `/v2/posts` (messy)
- **Rigid structure**: Once defined, hard to change without breaking clients

## GraphQL

### What is GraphQL?

GraphQL is a query language. Clients specify exactly what data they need. Server returns only that.

**Example: Blog API**

```graphql
query GetPost($id: ID!) {
  post(id: $id) {
    title
    content
    author {
      name
      email
    }
    comments {
      text
      createdAt
    }
  }
}
```

Client asks for title, content, author name/email, and comment text. Server returns **exactly** that—nothing more.

### GraphQL Example Code

**Backend (Node.js/Apollo)**:
```typescript
const typeDefs = gql`
  type Post {
    id: ID!
    title: String!
    content: String!
    author: User!
    comments: [Comment!]!
  }

  type User {
    id: ID!
    name: String!
    email: String!
  }

  type Comment {
    id: ID!
    text: String!
    createdAt: DateTime!
  }

  type Query {
    post(id: ID!): Post
    posts: [Post!]!
  }

  type Mutation {
    createPost(title: String!, content: String!): Post!
    updatePost(id: ID!, title: String, content: String): Post!
    deletePost(id: ID!): Boolean!
  }
`;

const resolvers = {
  Query: {
    post: (_, { id }) => db.post.findUnique({ where: { id } }),
    posts: () => db.post.findMany(),
  },
  Mutation: {
    createPost: (_, { title, content }) =>
      db.post.create({ data: { title, content } }),
  },
  Post: {
    author: (post) => db.user.findUnique({ where: { id: post.authorId } }),
    comments: (post) => db.comment.findMany({ where: { postId: post.id } }),
  },
};
```

**Frontend (React with Apollo)**:
```typescript
import { gql, useQuery } from '@apollo/client';

const GET_POST = gql`
  query GetPost($id: ID!) {
    post(id: $id) {
      title
      content
      author { name }
    }
  }
`;

export default function Post({ id }) {
  const { data, loading } = useQuery(GET_POST, { variables: { id } });

  if (loading) return <p>Loading...</p>;

  return (
    <article>
      <h1>{data.post.title}</h1>
      <p>By {data.post.author.name}</p>
      <p>{data.post.content}</p>
    </article>
  );
}
```

### GraphQL Pros

- **No over-fetching**: Get exactly what you ask for
- **No under-fetching**: Fetch related data in one query
- **Single endpoint**: `/graphql` handles everything
- **Introspective**: API describes itself; great for documentation
- **Versioning**: No versioning needed; evolve schema safely
- **Developer experience**: GraphQL IDE (Apollo Studio) with autocomplete

### GraphQL Cons

- **Complexity**: Learning curve; more setup required
- **Caching**: HTTP caching doesn't work; need custom caching
- **N+1 queries**: Easy to accidentally fetch too much data
- **File uploads**: More complex than REST
- **Monitoring**: Hard to track individual operations
- **Performance**: Can be slower if not optimized

## Comparison Table

| Aspect | REST | GraphQL |
|--------|------|---------|
| **Learning curve** | Easy | Medium-Hard |
| **Over-fetching** | Common | Not possible |

## Real-World Example: Switching from REST to GraphQL

### The Problem: Over/Under-Fetching

**REST endpoints:**
```
GET /api/users/123
  → Returns: { id, name, email, phone, address, bio, ... }
  
GET /api/users/123/posts
  → Returns: [{ id, title, content, likes, comments, ...}, ...]
```

Frontend needs user name and 5 recent posts.

**REST issue**: Request 1 returns too much (over-fetch). Request 2 might return 50 posts, but we want 5 (under-fetch).

**GraphQL solution**:
```graphql
query GetUserWithPosts($id: ID!) {
  user(id: $id) {
    name
    posts(first: 5) {
      id
      title
    }
  }
}
```

Returns exactly what's needed. One request. Minimal bandwidth.

## When to Use REST

- **Simple public APIs** (weather, quotes, public data)
- **Caching is critical** (HTTP caching works great)
- **Simple CRUD operations**
- **Legacy systems** (everyone knows REST)
- **File uploads** (easier than GraphQL)
- **High-traffic public APIs** (simpler to scale)

**Examples**: Twitter API (kind of), GitHub API (REST endpoints still available), Stripe

## When to Use GraphQL

- **Complex, interconnected data** (social networks, e-commerce)
- **Multiple client types** (web, mobile, TV) with different needs
- **Rapid development** (iterate without API versioning)
- **Data-heavy applications** (dashboards, analytics)
- **Internal APIs** (where you control both client and server)

**Examples**: GitHub (GraphQL endpoint now available), Shopify, Stripe (now has GraphQL)

## The Mixed Approach (2026 Standard)

Most companies use **both**:
- **REST** for simple, public, cacheable endpoints
- **GraphQL** for complex, internal, rapidly-changing APIs

Example architecture:
```
Public API (Simple) → REST
├─ GET /api/products
├─ GET /api/products/:id
└─ POST /api/products/:id/reviews

Internal API (Complex) → GraphQL
├─ query getUser($id: ID!) { user { ..., posts { ... } } }
├─ mutation createPost($title: String!) { createPost { ... } }
└─ subscription onPostCreated { post { ... } }
```

## Performance Comparison

### REST (N requests)
```
Client: GET /api/posts/1
        ↓
Server: DB query 1 (post) ✓
Server: DB query 2 (author) ✓
Server: DB query 3 (comments) ✓
        ↓
Client: Receives full payload

Time: ~50ms (with 3 DB queries) + network latency
```

### GraphQL (1 request, but N+1 risk)

```
Client: POST /graphql (query: post { id, author { name }, comments { text } })
        ↓
Server: DB query 1 (post) ✓
Server: DB query 2 (author) ← N+1 problem!
Server: DB query 3+ (comments) ← Might be N queries if not batched
        ↓
Client: Receives exactly what was requested

Time: Can be slower if not optimized (N+1 queries)
      Can be faster if batched correctly
```

**Optimization**: Use DataLoader to batch queries.

```typescript
const userLoader = new DataLoader(async (userIds) => {
  const users = await db.user.findMany({
    where: { id: { in: userIds } }
  });
  return userIds.map(id => users.find(u => u.id === id));
});

// Now: one batch DB query instead of N individual queries
```

## Summary

- **REST**: Simple, cacheable, good for public APIs
- **GraphQL**: Powerful, flexible, good for complex internal APIs
- **2026 trend**: Hybrid approach—REST for simple, GraphQL for complex

Choose based on your use case, not hype.
