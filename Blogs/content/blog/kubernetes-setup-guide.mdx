# How to Set Up a Kubernetes Cluster: Complete Guide for 2026

Kubernetes has become the standard for container orchestration in 2026. Every serious infrastructure team uses it. But setting it up can be overwhelming.

This guide walks you through setting up Kubernetes locally and in production, understanding what's happening at each step.

## What is Kubernetes?

**Kubernetes** (K8s) is a container orchestration platform that automates deployment, scaling, and management of containerized applications.

Instead of manually managing Docker containers on servers, Kubernetes:
- Runs containers across multiple machines (nodes)
- Scales up/down based on demand
- Handles failures (if a container crashes, Kubernetes restarts it)
- Manages networking between containers
- Handles storage persistence

## Kubernetes Architecture

Understanding the architecture helps you set it up correctly.

### Master Node (Control Plane)

The "brain" that makes decisions:
- **API Server**: Receives all commands
- **Scheduler**: Decides which node runs which container
- **Controller Manager**: Ensures actual state matches desired state
- **etcd**: Database storing all cluster state

### Worker Nodes

Machines that run containers:
- **Kubelet**: Agent ensuring containers run as specified
- **Runtime**: Docker or containerd
- **kube-proxy**: Handles networking between containers

### Communication

```
kubectl (your command)
    ↓
API Server
    ↓
Scheduler decides nodes
    ↓
Worker Node 1: Kubelet starts container
Worker Node 2: Kubelet starts container
```

## Option 1: Local Setup with Minikube

**Minikube** creates a single-node Kubernetes cluster locally. Perfect for learning and development.

### Install Minikube

**macOS (Apple Silicon/Intel):**
```bash
brew install minikube
```

**Windows:**
```bash
choco install minikube
```

**Linux:**
```bash
curl -LO https://github.com/kubernetes/minikube/releases/latest/download/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube
```

### Start a Cluster

```bash
minikube start
# Starting local Kubernetes cluster... DONE!
# kubectl is now configured to use minikube.
```

### Verify

```bash
kubectl get nodes
# NAME       STATUS   ROLES           AGE   VERSION
# minikube   Ready    control-plane   2m    v1.30.0
```

Success! You have a running cluster.

### Deploy Your First App

```bash
# Create a deployment
kubectl create deployment hello-world --image=nginx:latest

# Expose it with a service
kubectl expose deployment hello-world --type=LoadBalancer --port=80

# Get the access URL
minikube service hello-world --url
# http://127.0.0.1:53421

# Visit the URL in your browser
```

### Access Dashboard

```bash
minikube dashboard
# Opens a web UI showing cluster status, pods, services, etc.
```

## Option 2: Cloud Setup (AWS EKS)

Production setups live in cloud. AWS EKS (Elastic Kubernetes Service) is popular.

### Prerequisites

- AWS Account
- AWS CLI installed and configured
- `kubectl` installed
- `eksctl` (EKS CLI tool)

### Create a Cluster

```bash
eksctl create cluster \
  --name my-cluster \
  --region us-east-1 \
  --node-type t3.medium \
  --nodes 3
```

This:
- Creates a cluster named "my-cluster"
- In us-east-1 region
- With 3 worker nodes (t3.medium)
- Configures networking, security groups, and IAM roles automatically

Takes 10–15 minutes.

### Verify

```bash
kubectl get nodes
# NAME                          STATUS   ROLES    AGE   VERSION
# ip-10-0-45-12.ec2.internal   Ready    <none>   30s   v1.30.0
# ip-10-0-87-19.ec2.internal   Ready    <none>   40s   v1.30.0
# ip-10-0-156-45.ec2.internal  Ready    <none>   50s   v1.30.0
```

All three nodes are ready. Your cluster is live!

### Deploy an App

```yaml
# app.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
```

Deploy:
```bash
kubectl apply -f app.yaml

# Check status
kubectl get deployment
kubectl get service

# Get the external IP (may take 1–2 minutes)
kubectl get service nginx-service --watch
```

### Clean Up

```bash
eksctl delete cluster --name my-cluster --region us-east-1
```

## Key Kubernetes Concepts

### Deployment

Describes an app: how many replicas, which image, resource limits.

### Pod

Smallest unit in Kubernetes. Usually one container per pod (but can be multiple).

### Service

Exposes pods to the network. Types:
- **ClusterIP**: Only accessible within cluster
- **NodePort**: Accessible via any node IP + port
- **LoadBalancer**: Accessible via cloud provider's load balancer

### ConfigMap & Secret

Store configuration and sensitive data:

```yaml
# ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_URL: postgresql://db:5432/app
  LOG_LEVEL: info

---
# Secret
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  password: "your-secret-password"
```

Reference in pods:
```yaml
env:
- name: DATABASE_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-credentials
      key: password
```

### StatefulSet

Like Deployment but for stateful apps (databases). Guarantees pod identity and storage.

### Namespace

Virtual cluster inside a cluster. Isolate teams/environments:

```bash
kubectl create namespace staging
kubectl apply -f app.yaml -n staging
# The app runs in the staging namespace only
```

## Helm (Package Manager for K8s)

Helm simplifies Kubernetes deployments. Instead of writing YAML, use pre-made charts.

### Install Helm

```bash
brew install helm  # macOS
choco install kubernetes-helm  # Windows
```

### Using a Chart

```bash
# Add a repository
helm repo add bitnami https://charts.bitnami.com/bitnami

# Install PostgreSQL
helm install my-postgres bitnami/postgresql \
  --set auth.password=secretpass \
  --set primary.persistence.size=10Gi

# Verify
kubectl get pods
# my-postgres-0 is running
```

Much simpler than writing Deployment YAML manually.

## Monitoring & Logging

### Monitoring (Prometheus + Grafana)

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts

## Resources & Limits

Always specify resource requests and limits:

```yaml
containers:
- name: app
  resources:
    requests:
      cpu: "250m"
      memory: "256Mi"
    limits:
      cpu: "500m"
      memory: "512Mi"
```

- **requests**: What Kubernetes guarantees to the pod
- **limits**: Maximum the pod can use

Without limits, a rogue pod can crash the entire cluster.

## Auto-Scaling

### Horizontal Pod Autoscaler (HPA)

Automatically scale replicas based on CPU/memory:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

When CPU > 70%, Kubernetes spins up more replicas. When < 30%, it scales down.

## Best Practices

1. **Always set resource requests/limits**
2. **Use namespaces to isolate teams**
3. **Use ConfigMaps/Secrets, not hardcoded values**
4. **Implement readiness and liveness probes**:
   ```yaml
   livenessProbe:
     httpGet:
       path: /health
       port: 8080
     initialDelaySeconds: 10
     periodSeconds: 10
   ```
5. **Use RBAC for access control**
6. **Monitor everything** (CPU, memory, latency, errors)
7. **Backup your cluster state** (especially etcd)

## Troubleshooting

### Pod won't start
```bash
kubectl logs <pod-name>
kubectl describe pod <pod-name>
```

### Can't reach my service
```bash
kubectl get service my-service
# Check if it has an external IP

# If using LoadBalancer, IP might take 1–2 min to appear
kubectl get service my-service --watch
```

### Node is NotReady
```bash
kubectl describe node <node-name>
# Check CPU/memory pressure, disk space
```
